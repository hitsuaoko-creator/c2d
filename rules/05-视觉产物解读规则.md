# 视觉产物解读规则

Use this rule when the user provides output from the `visual-insight` skill or any visual analysis artifact.

## Position

The visual artifact is design strategy input, not a component specification. It helps decide why and how to generate the UI, while `rules/`, the MasterGo component library, and canvas hard rules remain binding.

## Required Summary

Before component mapping, summarize the visual artifact into `本次视觉生成概要` with:

1. 视觉目标：本次设计最重要的用户判断和业务目标。
2. 关键路径：每一步用户要看什么、判断什么、点击什么。
3. 信息层级：必须突出、正常呈现、需要弱化的内容。
4. 主 CTA：每个页面的主操作和次操作。
5. 状态补齐：loading、成功、失败、空状态、取消、回填、保存等。
6. 风险与误触：需要隔离、确认、弱化或提示的操作。
7. 组件策略：哪些区域应优先用组件库组件，哪些可能需要临时容器。
8. 可执行设计动作：能直接转成布局、层级、组件、状态或文案权重的动作。

## Conflict Handling

- If visual output conflicts with `rules/00-规则索引.md`, `rules/02-组件使用规则.md`, or `rules/03-画布与图层规则.md`, follow the c2d hard rules.
- If visual output conflicts with MasterGo component capabilities, use component capabilities and record the limitation.
- If visual output conflicts with the interaction screenshot, add `设计冲突说明` and ask when the conflict affects the core path.
- Do not invent components only because the visual output suggests an effect that the library does not support.

## Missing Visual Output

If no visual output is provided:

- Ask whether the user has a visual-insight artifact.
- If the user does not have one or wants to continue, proceed with a lightweight `本次视觉生成概要` based on screenshot, rules, and safe assumptions.
- Explicitly mark: `未提供 visual 产物，本次概要基于交互截图和 c2d rules 生成。`

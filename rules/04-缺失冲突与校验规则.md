# 缺失冲突与校验规则

目标：缺组件时不硬猜；规则冲突时不静默处理；提交前必须跑检查。

## 缺失组件策略

| 情况 | 策略 |
| --- | --- |
| 基础分组、背景、分割区缺组件 | 允许 `临时容器-用途` |
| 普通说明文案缺组件 | 允许普通文本节点 |
| 业务卡片缺匹配组件 | 先查 `封面卡片组件`、`首页卡片`、`图标卡片组件`、`点击卡片组件`、`上传卡片组件`；仍不匹配时询问 |
| 键盘缺失 | 停止，要求刷新组件库或确认替代方案 |
| 顶部导航缺失 | 停止，不能自由绘制替代完整导航 |
| tabbar 缺失 | 停止，不能贴底手绘替代 |
| 完整抽屉缺失 | 可用 `临时容器-抽屉` 承载 `抽屉顶部栏`、`drawer_input`、`option`、按钮组 |
| 图标库为空 | 先用 `icon` 组件；仍缺失时按页面生成规则用 FontAwesome 并说明 |
| props 不确定 | 不改写；选择明确枚举或询问 |
| slot 不存在 | 不伪造 slot；用临时容器或询问 |

## 禁止兜底

- 用临时 frame 替代 `键盘组件`。
- 用 `1:1 图片素材`、`3 纵列图片素材`、`placeholder_pic` 替代业务组件。
- 把组件实例拆成零件。
- 用内部结构组件拼出已有复合组件。
- 伪造 props、slots、instance_swap。
- 把流程箭头、红框、说明文字放进手机 Frame。

## 冲突优先级

1. 用户当前明确要求。
2. MasterGo 组件 JSON 的真实 `name`、`props`、`slots`。
3. MasterGo 变量 JSON 的真实变量名。
4. 本 rules 文档。
5. AI 基于截图和命名的推断。

## 必须写入设计稿的冲突

出现以下任一情况，在手机 Frame 外新增 `设计冲突说明`：

- 用户要求与组件库能力冲突。
- 组件名暗示用途，但 props 或 slot 不支持。
- 颜色要求与变量库不一致。
- 浅色和深色变量无法对应。
- 用户要求使用素材，但规则禁止素材作为业务组件入口。
- 键盘、顶部导航、tabbar、业务卡片缺失。
- 画板被生成为自动布局。
- `一级界面顶部渐变` 缺失或不在 `y=0`。
- 状态栏和顶部导航栏之间出现空隙。
- tabbar 未悬浮或底部安全距离不是 `16dp`。

模板：

```md
## 设计冲突说明

- 冲突编号：
- 冲突类型：组件 / 变量 / 颜色 / 文案 / 交互 / 画布
- 涉及对象：
- 用户需求：
- 组件或变量事实：
- 冲突点：
- 本次处理：
- 需要确认：
```

## 提交前 hook

提交到 MasterGo 前必须运行：

```bash
scripts/check_mastergo_html.py <html-file> --library <MasterGo组件库快照目录>
```

有 `ERROR` 时不能提交。`WARN` 需要人工确认并在提交前报告中说明。

## 提交前报告

调用 `submit_page_to_canvas` 前，简短报告：

- `交互截图解读`
- `本次视觉生成概要`，如有 visual-insight 产物
- `组件候选清单`
- `临时容器清单`
- `设计冲突说明`
- `hook 结果`
